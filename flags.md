- **21. ~~Playwright E2E for the UI~~** ✅ [ui-tests/](ui-tests/) — 23 tests in three specs (auth / accounts / admin) run through real Chromium against the live stack in ~28s. HTML report, screenshots, video, and trace-viewer captured on failure. Writing the suite surfaced two real race bugs, both now fixed:
  1. **Register→login race in the auth backend.** FastAPI's yield-dep teardown commits the DB transaction *after* the response is delivered — so a fast client that fires /login right after /register saw a 401 because the INSERT hadn't been committed. Real-user impact: intermittent "invalid credentials" right after signup. Fix: moved `with main_conn.transaction()` out of `deps_factory` and into each write route's handler ([auth_service/src/auth_service/application/deps.py](auth_service/src/auth_service/application/deps.py) now exposes `transaction`, [auth_service/src/auth_service/infrastructure/app.py](auth_service/src/auth_service/infrastructure/app.py) wraps register/login/refresh + bootstrap in `with deps.transaction():`). Commit now happens before response send.
  2. **Refresh-rotation race in the UI.** app.js eagerly called `/refresh` on boot, rotating the login-provided refresh token. A reload during that in-flight rotation left `sessionStorage` with a token that had just been revoked → next boot's refresh 401'd → logout. Real-user impact: reloading within ~100ms of login logs you out. Fix: persist the (short-lived, 5-min) access token to `sessionStorage` too and skip the eager refresh when it's present; api()'s existing 401-retry-with-refresh path handles genuine expiry.
- **22. ~~Rate limit too tight for interactive UI~~** ✅ Bumped `rate_limit` events from 60/min to 300/min in [proxy/caddy/Caddyfile](proxy/caddy/Caddyfile). 60/min was tripping during a normal Playwright E2E run and would trip real users too — a customer clicking through accounts + transfers with no browser cache can easily produce 60 requests in a minute. 300/min still trips bruteforce loops (which do thousands per minute) and stops most credential-stuffing patterns.

### 13. Client IP preservation through Docker bridge NAT
When traffic enters Caddy from the host via Docker's default bridge network, the source IP Caddy sees is the Docker gateway (e.g. `172.19.0.1`), not the real client's IP. The value ends up in the fail2ban log line, so bans would apply to the gateway rather than the actual attacker. Fine for a demo; production would either:
- run Caddy with `network_mode: host`, or
- deploy behind a real load balancer that forwards a trusted `X-Forwarded-For` chain (auth's `_client_ip` helper already prefers `X-Real-IP` set by Caddy).
- File: [auth_service/src/auth_service/infrastructure/app.py](auth_service/src/auth_service/infrastructure/app.py) `_client_ip`
